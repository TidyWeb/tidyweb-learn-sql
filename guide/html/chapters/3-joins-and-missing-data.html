Joins And Missing Data

Real databases split related facts across tables so the same information does not have to be repeated everywhere. A join reconnects those facts for one answer. The hard part is not typing JOIN; it is identifying the relationship and deciding what should happen when a match is missing.

Why The Data Is Split

One fact, one home

The floor belongs to a room, not separately to every object. If twenty objects share the Navigation room, storing its floor twenty times creates repetition and twenty places that could become inconsistent.

Keys create reliable links

A primary key uniquely identifies a row in its own table. A foreign key stores that identifier in a related table. Here museum_rooms.room_id identifies a room, and museum_objects.room_id points towards it.

The Relationship Sentence

Before writing a join, say the link in ordinary language:

“Match each museum object to the museum room whose room_id is equal to the object’s room_id.”

If you cannot say which columns should match and why, you are not ready to write the ON clause.

Concept 1 · Matching records only

INNER JOIN Keeps Successful Matches

An inner join considers each row from the first table and looks for a row in the joined table that satisfies the ON rule. Only successful pairs reach the result.

Command bridge
FROM museum_objects INNER JOIN museum_rooms ON museum_objects.room_id = museum_rooms.room_id

INNER JOIN museum_roomsIntroduces the second table and says unmatched rows should not survive.

museum_rooms.room_idThe related identifier in the second table.

FROM museum_objects

A Join Does Not Automatically Mean “Everything”

Wrong instinct: “Combining the tables” sounds as though every record from both must appear.

Correction: an inner join means the intersection of successful matches. If the question says every object must remain, the join choice must express that requirement.

Concept 2 · Preserve one side

LEFT JOIN Keeps Every Row From The Left Table

“Left” means the table written before LEFT JOIN. Every row from that table survives. When there is no matching row on the right, SQL supplies NULL for the missing right-side columns.

Worked example · left join result
SELECT museum_objects.object_name,
       museum_rooms.room_name
FROM museum_objects
LEFT JOIN museum_rooms
  ON museum_objects.room_id = museum_rooms.room_id;

Inner Join Question

Use it when the wording means: “show records that have a related record.” Unmatched rows should not be part of the answer.

Left Join Question

Use it when the wording means: “show every record from this main table, with related information where available.” The table you must preserve belongs on the left.

Qualified Column Names And Aliases

Both tables contain room_id, so writing only room_id can be ambiguous. The full form is clear but long. A table alias is a temporary short name used only inside this query.

SELECT o.object_name, r.room_name
FROM museum_objects AS o
LEFT JOIN museum_rooms AS r
  ON o.room_id = r.room_id;
AS oTemporarily calls museum_objects by the shorter name o.
o.object_nameMeans the object_name column from the table currently aliased as o.
AS rTemporarily calls museum_rooms by the shorter name r.

An alias does not rename the stored table. It only makes this statement easier to read.

Concept 3 · Missing is not an ordinary value

NULL Means “No Known Value Here”

NULL is not zero, an empty text value, the word “NULL”, or automatically an error. It represents absence or unknown information. Because it is not an ordinary value, ordinary comparisons such as = NULL do not ask the right question.

Command bridge
IS NULLKeep rows where the value is missing.
IS NOT NULLKeep rows where a value is present.
WHERE room_id IS NULL
WHERE room_id IS NOT NULL
Worked example · find the missing relationship
SELECT o.object_name
FROM museum_objects AS o
LEFT JOIN museum_rooms AS r
  ON o.room_id = r.room_id
WHERE r.room_id IS NULL;

This pattern asks: keep every object, then retain only those for which the joined room side failed to appear.

Five Questions Before Syntax

  1. What should one result row represent?
  2. Which table contains the main records the question is about?
  3. Which table contains the extra facts?
  4. Which identifier connects those tables, and does it represent the same thing on both sides?
  5. Should unmatched main records disappear or remain?
