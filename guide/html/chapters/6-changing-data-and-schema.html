Chapter 6 · Controlled Change

Changing Data And Schema

Reading SQL asks the database for an answer. Writing SQL changes the database that later answers come from. This chapter separates row changes from structure changes, explains every statement as a before-and-after operation, and makes the safety boundary part of the syntax—not an afterthought.

Data Manipulation

These statements change rows inside an existing table.

  • INSERT INTO adds rows.
  • UPDATE changes values in existing rows.
  • DELETE FROM removes rows.

Data Definition

These statements change database structure, also called the schema.

  • CREATE TABLE defines a new table.
  • ALTER TABLE changes an existing table definition.
  • DROP TABLE removes the table and its rows.
The non-negotiable safety routine

Preview → Change → Verify

  1. Preview: run a SELECT with the intended WHERE clause. Inspect which rows it finds.
  2. Change: keep the proven boundary and replace the reading statement with UPDATE or DELETE.
  3. Verify: query the affected rows or schema again. Confirm both what changed and what did not.

On a real database, transactions, permissions, backups, and review provide further protection. The small resettable lab is a safe place to learn the statement shapes; it is not a reason to skip those protections in live work.

Concept 1 · Add a complete record

INSERT INTO Maps Columns To Values

An insert creates a new row. The safest readable form names the destination columns, then supplies values in exactly the same order.

Command bridge
INSERT INTO instruments (instrument_id, instrument_name, family, hourly_fee, available) VALUES (5, 'Acoustic Guitar', 'Strings', 12, 'Y');
INSERT INTOStarts a row-addition statement.
instrumentsThe destination table.
(column list)The fields being filled, written in the order expected from the values.
VALUESIntroduces the new row data.
(value list)One value for each named column, in matching position. Text is quoted; numbers are bare.
Worked example · positional mapping
PositionColumnValue
1instrument_id5
2instrument_name'Acoustic Guitar'
3family'Strings'

After the statement, a fifth row exists. None of the previous four rows is changed.

Why Naming Columns Is Worth The Extra Words

If you omit the column list, the values must match the table’s full stored column order. That makes the statement fragile when the schema is unfamiliar or later changes. A named list makes the mapping visible and allows omitted columns to receive permitted defaults or NULL values.

Concept 2 · Change existing values

UPDATE Uses SET For The New Value And WHERE For The Boundary

SET answers “what should change?” WHERE answers “in which rows?” These are separate decisions. The right calculation with the wrong boundary is still a dangerous query.

Command bridge
UPDATE instruments SET hourly_fee = hourly_fee + 2 WHERE family = 'Strings';
UPDATE instrumentsNames the table whose existing rows may change.
SET hourly_fee =Names the column receiving a new value.
hourly_fee + 2Uses each row’s current value and adds 2. It is an increase, not a replacement with the number 2.
WHERE family = 'Strings'Limits the change to rows whose family is exactly Strings.
Worked example · before and after

Preview first

SELECT instrument_id, instrument_name, hourly_fee
FROM instruments
WHERE family = 'Strings';

Without WHERE, Every Row Is In Scope

UPDATE instruments
SET available = 'N';

This is valid SQL. It deliberately changes every row. The danger is not that the database fails; it is that the database does exactly what was written. Treat a missing WHERE as an explicit “all rows” decision that must be intentional.

Concept 3 · Remove records

DELETE FROM Removes Whole Rows

A delete does not clear just the named condition column. Every column in each matching row disappears because the row itself is removed.

Worked example · prove the target
SELECT *
FROM instruments
WHERE instrument_id = 4
  AND instrument_name = 'Tenor Saxophone';

DELETE FROM instruments
WHERE instrument_id = 4
  AND instrument_name = 'Tenor Saxophone';

The preview and delete use the same two-part boundary. Requiring both the unique identifier and expected name makes the intended record especially visible while learning.

DELETE, Emptying A Value, And DROP Are Different

ActionWhat remains
UPDATE ... SET available = NULLThe row and table remain; one value becomes missing.
DELETE FROM instruments WHERE ...The selected rows go; the table and other rows remain.
DROP TABLE instrumentsThe table definition and all its rows go.
Concept 4 · Define a table

A Schema Describes What Rows Are Allowed To Look Like

CREATE TABLE names columns, assigns data types, and can attach constraints—rules the database enforces. This is structure, not row data.

Command bridge
CREATE TABLE bookings (
  booking_id INTEGER PRIMARY KEY,
  instrument_id INTEGER NOT NULL,
  booked_by TEXT NOT NULL,
  hours INTEGER DEFAULT 1
);
CREATE TABLE bookingsCreates a new table named bookings.
INTEGERA whole-number data type.
TEXTA text data type.
PRIMARY KEYRequires the main identifier to uniquely identify each row.
NOT NULLRequires a value; missingness is not allowed in that column.
DEFAULT 1Uses 1 when an insert legitimately omits the hours value.
commasSeparate one column definition from the next inside the parentheses.

Data Types And Constraints Answer Different Questions

  • A data type asks: what kind of value belongs here—whole number, decimal, text, date, and so on?
  • A constraint asks: what rule must this value or row obey—unique, required, within a range, linked to another table?
  • A good schema prevents impossible data instead of relying on every future query writer to remember every rule.

ALTER TABLE Changes Structure

ALTER TABLE instruments
ADD storage_zone TEXT;

This adds a column definition to the existing table. Existing rows remain. In many databases their new storage_zone value begins as NULL unless a default or migration supplies something else.

DROP TABLE Removes Structure

DROP TABLE old_bookings;

The table and its data are removed. IF EXISTS can prevent an error when the table is already absent, but it does not make dropping the wrong existing table safe.

A reliable change-planning routine

State The Blast Radius Before Running

  1. Name the object being changed: rows in which table, or which table definition?
  2. State the intended before-and-after result in one sentence.
  3. For UPDATE or DELETE, preview the exact WHERE boundary with SELECT.
  4. Check whether the statement affects one row, several rows, all rows, one column definition, or an entire table.
  5. Run the smallest complete statement.
  6. Verify the changed rows or schema, including nearby data that should remain untouched.

Frequent Change-Query Mistakes

Wrong turnRepair
Insert columns and values do not line up.Pair them by position before running; count both lists.

An update or delete boundary is guessed.Run the same boundary as a SELECT and inspect the returned rows first.
