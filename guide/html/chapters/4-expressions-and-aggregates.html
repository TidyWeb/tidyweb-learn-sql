Expressions And Aggregates

Until now, the result has mostly repeated facts already stored in a table. Expressions let SQL calculate a new value for each row. Aggregates go further: they turn many rows into one summary—or one summary per group. The central question becomes: what should one result row represent?

01Calculate per rowBuild expressions from existing values and label them with AS.
02Summarise all rowsUse COUNT, SUM, AVG, MIN, and MAX.
03Create groupsUse GROUP BY to change the result grain.
04Filter at the right levelUse WHERE for source rows and HAVING for finished groups.

Concept 1 · A new value for every source row

An Expression Calculates Without Changing Stored Data

An expression combines columns, literal values, operators, or functions to produce a value in the result. SQL evaluates it separately for each source row. The calculation is temporary unless a separate data-changing statement stores it.

Command bridge
birds_seen / hours AS birds_per_hour
birds_seenThe stored bird count for the current row.
/The division operator.
hoursThe stored survey duration for the same row.
ASIntroduces a temporary result-column name, called an alias.
birds_per_hourThe readable label chosen for the calculated output. It is not a stored column in the source table.
Worked example · expression result
SELECT site,
       birds_seen,
       hours,
       birds_seen * 1.0 / hours AS birds_per_hour
FROM surveys;
sitebirds_seenhoursbirds_per_hour
Reed Pool1829.0
Oak Rise933.0
Marsh Edge2446.0
Pine Track1226.0
Cliff Path1535.0

The * 1.0 encourages decimal division in systems where dividing whole numbers could otherwise produce a whole-number result.

An Alias Is A Label, Not The Calculation

Wrong instinct: think AS birds_per_hour performs the maths.

Correction: birds_seen * 1.0 / hours performs the calculation. AS merely gives the returned expression a clear heading so later people do not have to decode it.

Concept 2 · Many rows become one answer

Aggregate Functions Summarise A Set Of Rows

An aggregate function accepts values from several rows and returns one summary value. Without GROUP BY, the whole filtered table is treated as one set.

The five foundation aggregates
COUNT(*)Counts result rows, including rows that contain missing values in some columns.
COUNT(column)Counts non-NULL values in that particular column.
SUM(column)Adds the non-missing numeric values.
AVG(column)Calculates the mean of the non-missing numeric values.
MIN(column)Returns the smallest non-missing value.
MAX(column)Returns the largest non-missing value.
Worked example · one summary row
SELECT COUNT(*) AS survey_count,
       SUM(birds_seen) AS total_birds,
       AVG(hours) AS average_hours,
       MAX(birds_seen) AS highest_count
FROM surveys;
survey_counttotal_birdsaverage_hourshighest_count
5782.824

Five source rows have been compressed into one result row containing four different summaries.

The Grain Question

Grain means what one row represents. Before aggregation, one row represents one survey visit. After the query above, one row represents the entire set of survey visits.

If you cannot finish the sentence “one result row will represent…”, pause before adding GROUP BY.

Concept 3 · One summary for each category

GROUP BY Splits Rows Into Buckets

GROUP BY habitat gathers all Wetland rows into one bucket, all Woodland rows into another, and so on. The aggregate functions then calculate separately inside each bucket.

Command bridge
SELECT habitat, SUM(birds_seen) ... GROUP BY habitat;
habitatThe category label shown once for each group.
SUM(birds_seen)The calculation repeated independently inside every habitat group.
GROUP BY habitatChanges the grain to one result row per distinct habitat.
Worked example · grouped result
SELECT habitat,
       COUNT(*) AS survey_count,
       SUM(birds_seen) AS total_birds
FROM surveys
GROUP BY habitat;
habitatsurvey_counttotal_birds
Coastal115
Wetland242
Woodland221

The five visit rows have become three habitat rows.

Why Un-grouped Detail Columns Do Not Belong

SELECT habitat, observer, SUM(birds_seen)
FROM surveys
GROUP BY habitat;

One habitat group can contain several observers. SQL cannot honestly place one arbitrary observer beside the habitat total and pretend that person represents the whole group. The safe beginner rule is: every selected item should either be a grouped column or an aggregate calculation.

Grouping By More Than One Column

Each additional grouped column makes the grain more detailed.

SELECT habitat,
       observer,
       SUM(birds_seen) AS total_birds
FROM surveys
GROUP BY habitat, observer;

One result row now represents one unique habitat-and-observer combination, not habitat alone.

Concept 4 · Filter before or after grouping

WHERE And HAVING Work At Different Levels

WHERE decides which individual source rows are allowed into the grouping stage. HAVING decides which completed group summaries survive afterwards.

Worked example · two filters, two jobs
SELECT habitat,
       SUM(birds_seen) AS total_birds
FROM surveys
WHERE hours >= 3
GROUP BY habitat
HAVING SUM(birds_seen) > 10;
  1. WHERE hours >= 3 removes the shorter individual visits.
  2. GROUP BY habitat creates groups from the remaining rows.
  3. SUM(birds_seen) calculates a total for each group.
  4. HAVING SUM(birds_seen) > 10 removes completed groups whose total is not above 10.

Use WHERE When…

  • The condition is about a stored value in one source row.
  • You want to exclude rows before the calculation.
  • The wording says “only visits lasting at least three hours”.

Use HAVING When…

  • The condition is about a finished group or aggregate.
  • You want to exclude groups after summary values exist.
  • The wording says “only habitats with more than ten birds”.
A reliable aggregation planning routine

Write The Meaning Before The Query

  1. Complete: “one source row represents…”
  2. Decide whether the question wants detail rows, one overall summary, or one summary per category.
  3. If grouped, complete: “one result row represents…”
  4. Identify filters about individual rows; place those in WHERE.
  5. Identify filters about group calculations; place those in HAVING.
  6. Name every calculated column so the output explains itself.

Common Analytical Wrong Turns

Wrong turnRepair
Expecting an expression to reduce the row count.A row-level expression usually adds a calculated value to every row. Aggregation is what compresses rows.
Using WHERE COUNT(*) > 2.The count does not exist during WHERE. Group first, then use HAVING COUNT(*) > 2.
Selecting a detail column that is neither grouped nor aggregated.Remove it, aggregate it, or add it to GROUP BY only if the more detailed grain is genuinely intended.
Assuming COUNT(column) and COUNT(*) always match.Remember that COUNT(column) ignores missing values in that column.
