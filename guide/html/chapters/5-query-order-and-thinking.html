Chapter 5 · Deeper Reasoning

Query Order And Thinking

SQL is written in an order that reads well to a person, but the database builds the answer in a different logical order. Understanding that hidden journey turns several “odd rules” into predictable consequences. It also prepares you to solve problems in stages with subqueries and to stack compatible results with set operations.

The Written Order

This is the order in which clauses normally appear in a statement:

SELECT
FROM / JOIN
WHERE
GROUP BY
HAVING
ORDER BY
LIMIT / OFFSET

It leads with the desired output, which makes the question readable.

The Logical Order

This is a useful beginner model of how the answer is assembled:

FROM / JOIN
WHERE
GROUP BY
HAVING
SELECT
DISTINCT
ORDER BY
LIMIT / OFFSET

Database engines may optimise the physical work, but the result must behave as though these logical stages occurred.

Why This Model Earns Its Place

  • WHERE cannot filter a group total because grouping has not happened yet.
  • HAVING can filter a group total because groups now exist.
  • A SELECT alias often cannot be used in WHERE because the alias has not been created at that stage.
  • ORDER BY can often use a SELECT alias because selection has already named the output.
  • LIMIT comes last because SQL must first know which rows are at the front.
Concept 1 · Trace the answer through stages

One Question, Several Transformations

Question: Among courses costing at least 40, show each format’s average enrolment when that average is at least 14; highest average first.

SELECT format,
       AVG(enrolled) AS average_enrolment
FROM courses
WHERE fee >= 40
GROUP BY format
HAVING AVG(enrolled) >= 14
ORDER BY average_enrolment DESC;
Worked example · watch the result take shape
  1. FROM courses
    Start with all six source rows.
  2. WHERE fee >= 40
    Keep Woodwork Basics, Digital Photos, Bread Making, and Bike Repair. The 35 and 25 fee rows leave now.
  3. GROUP BY format
    Create an Online group and a Workshop group from the surviving rows.
  4. HAVING AVG(enrolled) >= 14
    Online averages 30. Workshop averages 14.67. Both groups survive.
  5. SELECT ... AS average_enrolment
    Build the two requested output columns and give the calculation a readable heading.
  6. ORDER BY average_enrolment DESC
    Place the higher finished average first.

Do Not Memorise The List Without Using It

The point is not to recite eight clause names. Use the order as a diagnostic tool: “Does the value I am referring to exist yet?” If not, the condition belongs later, the expression must be repeated, or the current query needs to become an inner stage.

Concept 2 · An intermediate answer

A Subquery Is A Query Used By Another Query

Some questions compare each row with a fact that must first be calculated from the table itself. A subquery creates that intermediate answer. Parentheses mark where the inner query begins and ends.

Command bridge
WHERE fee > (SELECT AVG(fee) FROM courses)
SELECT AVG(fee)The inner query calculates one value: the overall average fee.
( ... )Parentheses contain the inner query and make its result available as one expression.
fee >The outer query compares each course row with that returned average.
Worked example · inner first, outer second
SELECT course_name, fee
FROM courses
WHERE fee > (
  SELECT AVG(fee)
  FROM courses
);
  1. The inner query calculates the overall average fee: 52.5.
  2. The outer query tests each row’s fee against 52.5.
  3. Only fees above 52.5 remain.

Know The Shape The Inner Query Returns

The comparison fee > (...) expects one scalar value—a single cell. If the inner query returns several rows, ordinary > does not know which one to use. Before writing the outer query, ask whether the inner answer is one value, one column of values, or a whole table. The outer operator must match that shape.

Concept 3 · Put answer sets one above another

Set Operations Stack Rows Vertically

A join connects related columns side by side. A set operation combines the rows returned by complete queries. Each side must return the same number of columns in the same order, with compatible kinds of values.

Command bridge
UNIONStacks the two result sets and removes duplicate result rows.
UNION ALLStacks the two result sets and keeps every row, including duplicates.
SELECT course_name FROM spring_sessions
UNION
SELECT course_name FROM autumn_sessions;

Join, Subquery, Or Union?

If the question needs…Think first about…
Extra columns from related recordsA join: connect tables side by side through a key.
An intermediate value or set needed by another conditionA subquery: calculate the inner answer, then use it outside.
Compatible result rows from separate queries in one column layoutA union: stack the answers vertically.

Frequent Deeper-Query Mistakes

Wrong turnRepair
Using a selected aggregate alias inside WHERE.The alias and aggregate do not exist yet. Filter source rows with WHERE or group results with HAVING.
Writing the outer query before knowing the inner query’s result.Run or reason through the inner query alone first. State its shape and value.
Using a join to place two similar lists together.If the desired columns line up and the rows should stack, use a set operation.
Using UNION when duplicate occurrences carry meaning.Choose UNION ALL when every occurrence must remain.
Sorting each side of a union independently.Build the combined result first, then apply one final ORDER BY to the completed set.
