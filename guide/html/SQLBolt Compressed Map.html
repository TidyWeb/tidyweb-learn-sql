SQLBolt Learner's Guide

This page is a visual restructuring of the SQLBolt lesson sequence. It is not a copy of the tutorial. The goal is to show the course as a framework you can see at a glance: the main phases, the lesson purpose inside each phase, and the points where beginner understanding usually bends or breaks.

Original SQLBolt tutorial https://sqlbolt.com/

How To Use This

  • Look at the roadmap before opening a lesson.
  • Ask the lesson’s question in plain English first.
  • Use the lesson card to see the core SQL shape and the real concept change.
  • Slow down at the marked difficulty jumps instead of treating every lesson as equal.

Quick Syntax Glossary

;
End of query. Like pressing full stop on the SQL statement. In many teaching tools it is optional for one query, but in real SQL you should treat it as the normal ending.
,
Separator. It separates columns or values, like listing items: title, year.
'text'
Text value. Strings are quoted in SQL. If it is a word value, quote it. Numbers do not need quotes.
*
All columns. In SELECT *, the star means return every column from the table. It is not the text-search wildcard.
=
Equals test. SQL uses a single equals sign in filters: WHERE year = 2003. That is closer to Python == than Python =.
%
Text wildcard in LIKE. It means any run of characters inside a text pattern, not all columns and not Python modulo in this context.

Course Roadmap

Read One Table Shape The Answer Connect Tables Summarise Change Data Safely

Phase 1: Read One Table

The first job is learning that SQL is a question language. You ask for columns, then keep only the rows that matter. This is where the habit of phrasing the question clearly matters most.

Foundation

Introduction

Start

Relational tables, rows, columns, and the idea that a query is a request for a shaped answer.

  • Main point: think "what question am I asking the data?"
  • Key moment: a result table is the answer to that question.

Lesson 1: SELECT queries 101

01

Choose which columns you want from a single table.

  • Core shape: `SELECT ... FROM ...`
  • Key moment: stop saying "show the table"; start saying "return these fields."

Lesson 2: Constraints (Pt. 1)

02

Filter rows with numerical conditions, ranges, and lists.

  • Core shape: `WHERE`
  • Key moment: row-by-row filtering becomes part of the question.

Lesson 3: Constraints (Pt. 2)

03

Text filtering, quoted strings, exact matches, and pattern matching.

  • Core shape: `LIKE`, `IN`, quoted text
  • Key moment: "contains", "starts with", and "equals" are different jobs.

Phase 2: Shape The Answer

Once the right rows are in play, you control how the answer appears. This is still single-table thinking, but with more deliberate output shaping.

Output Control

Lesson 4: Filtering and sorting

04

Remove duplicates, order the result, and keep only the slice you need.

  • Core shape: `DISTINCT`, `ORDER BY`, `LIMIT`, `OFFSET`
  • Key moment: sorting is part of the answer, not decoration.

Review: Simple SELECT queries

05

The first real checkpoint: translate a word problem into a query without being led line by line.

  • Main point: combine earlier skills.
  • Key moment: test whether you can think in query pieces independently.

Phase 3: Connect Tables

This is where SQL stops feeling like spreadsheet filtering and starts feeling like database work. The important leap is understanding keys and matching logic.

Relationships

Lesson 6: Multi-table queries with JOINs

06

Combine data from two tables using a shared key.

  • Core shape: `INNER JOIN ... ON ...`
  • Key moment: if the `ON` match is wrong, the entire answer is wrong.

Lesson 7: OUTER JOINs

07

Keep unmatched rows when missing relationships matter.

  • Core shape: `LEFT JOIN`, `RIGHT JOIN`, `FULL JOIN`
  • Key moment: ask whether you want only matches or the missing edges too.

Lesson 8: NULLs

08

Missing data is not an ordinary value and needs special tests.

  • Core shape: `IS NULL`, `IS NOT NULL`
  • Key moment: blank-looking data changes query logic.

Lesson 9: Expressions

09

Build new output values with arithmetic, functions, and aliases.

  • Main point: SQL can compute, not just fetch.
  • Key moment: useful output is often constructed, not merely retrieved.

Phase 4: Summarise And Reason

This is the major conceptual jump. You stop thinking only in individual rows and start thinking in grouped categories, summaries, and logical processing order.

Analysis

Lesson 10: Aggregates (Pt. 1)

10

Summarise many rows into counts, sums, averages, minimums, and maximums.

  • Core shape: `COUNT`, `SUM`, `AVG`, `MIN`, `MAX`
  • Key moment: shift from "what rows?" to "how much?" and "how many?"

Lesson 11: Aggregates (Pt. 2)

11

Group rows, then filter the groups after they exist.

  • Core shape: `GROUP BY ... HAVING ...`
  • Key moment: `WHERE` is before grouping, `HAVING` is after grouping.

Lesson 12: Order of execution

12

Understand why SQL is written in one order but processed in another.

  • Main point: logic order explains earlier confusion.
  • Key moment: clause visibility depends on when the database evaluates it.

Topics: Subqueries and Set Operations

Extra

Handle intermediate answers and combine compatible results as sets.

  • Core shapes: nested queries, `UNION`, `UNION ALL`
  • Key moment: joins place tables side by side; unions stack results vertically.

Phase 5: Change Data And Structure

The final block moves from reading to editing. The syntax is not the hard part here. The hard part is learning to respect the blast radius.

Safety

Lesson 13: Inserting rows

13

Add new records that fit the table schema.

  • Core shape: `INSERT INTO ... VALUES ...`
  • Key moment: schema controls what valid data looks like.

Lesson 14: Updating rows

14

Change existing data with deliberate constraints.

  • Core shape: `UPDATE ... SET ... WHERE ...`
  • Key moment: `WHERE` is safety equipment, not decoration.

Lesson 15: Deleting rows

15

Remove rows without wiping the table by accident.

  • Core shape: `DELETE FROM ... WHERE ...`
  • Key moment: deleting rows is destructive filtering.

Lessons 16 to 18: Schema changes

16-18

Create, alter, and remove tables as structural database actions.

  • Core shapes: `CREATE TABLE`, `ALTER TABLE`, `DROP TABLE`
  • Key moment: deleting rows and deleting the table are completely different stakes.

Full Syntax Glossary

Syntax Only Short Definitions Python-Brain Friendly

Selecting And Naming

The pieces that ask for columns and name the result you want back.

SELECT
Choose columns. This starts the query and says what you want returned.SELECT title, year
FROM
Name the table. This tells SQL where to get the data from.FROM movies
*
All columns. In SELECT *, return every column from the table.SELECT * FROM movies;
DISTINCT
Remove duplicate rows. Use this when repeated results should collapse to unique ones.SELECT DISTINCT director FROM movies;
AS
Rename a result. Like a label for a returned column or table alias.year AS release_year

Filtering Rows

The syntax that decides which rows stay and which rows disappear.

WHERE
Filter rows. Only rows where the condition is true remain.WHERE year = 2004
=
Equals test. In SQL filters, single = behaves like Python ==.WHERE club = 'Liverpool'
!=
Not equal. Some databases also allow <> for the same job.WHERE club != 'Liverpool'
> < >= <=
Comparison tests. Greater than, less than, and their inclusive versions.WHERE goals >= 20
AND / OR
Combine conditions. AND means both must be true. OR means either can be true.WHERE club = 'Arsenal' AND goals > 10
BETWEEN
Inclusive range. BETWEEN 2000 AND 2010 includes both ends.WHERE year BETWEEN 2000 AND 2010
IN (...)
Value is in this list. Useful when checking against several allowed options.WHERE id IN (1, 3, 5)
NOT IN
Value is not in this list. The exclusion version of IN.WHERE club NOT IN ('Arsenal', 'Liverpool')

Text Matching

The extra syntax that appears when you are matching words rather than just numbers.

'text'
Quoted string. Text values are written in quotes so SQL knows they are text.'John Lasseter'
LIKE
Pattern match. Use this when you are matching a text shape, not just exact text.WHERE title LIKE 'Toy Story%'
NOT LIKE
Pattern does not match. The exclusion version of LIKE.WHERE title NOT LIKE 'Toy%'
%
Any run of characters. Only a wildcard inside LIKE patterns.'WALL-%'
_
Exactly one character. Also only used inside LIKE patterns.'WALL-_'

Sorting And Slicing

The syntax that shapes the order of the answer and trims how much you see.

ORDER BY
Sort the result. This changes the order of the returned rows.ORDER BY year
ASC
Ascending. Small to large, A to Z. Usually the default.ORDER BY year ASC
DESC
Descending. Large to small, Z to A.ORDER BY year DESC
LIMIT
Only this many rows. Keep the first n returned rows.LIMIT 5
OFFSET
Skip rows first. Often used with LIMIT to step through results.LIMIT 5 OFFSET 10

Joins And Missing Data

The syntax for combining tables and dealing with rows that do not match cleanly.

JOIN
Combine tables. Bring related rows together from more than one table.
INNER JOIN
Matches only. Keep rows that match in both tables.
LEFT JOIN
Keep the left table. Show all left-side rows, even when the right side has no match.
RIGHT JOIN
Keep the right table. The mirror version of LEFT JOIN.
FULL JOIN
Keep both sides. Show all rows from both tables, matched where possible.
ON
Match rule for the join. This says which columns connect the two tables.
NULL
Missing value. Not the same thing as zero or empty text.
IS NULL
Check for missing. Use this instead of = NULL.
IS NOT NULL
Check for present. Keep rows where the value is not missing.

Calculations And Summaries

The syntax for computed columns, grouped answers, and aggregate thinking.

expr AS name
Calculated column with a label. Build a value during the query and name it.
COUNT()
Count rows. Often used to answer "how many?"
SUM()
Add values. Total a numeric column.
AVG()
Average values. Mean of a numeric column.
MIN() / MAX()
Smallest or largest value. Useful for earliest, latest, lowest, highest.
GROUP BY
Group rows by category. This turns row-level data into grouped summaries.
HAVING
Filter groups. Like WHERE, but after the grouping has happened.

Changing Data

The statements that stop reading and start editing table contents.

INSERT INTO
Add new rows. Put new records into a table.
VALUES (...)
The new data. The values being inserted for the row or rows.
UPDATE
Change existing rows. Edit rows already in the table.
SET
New column values. Used inside UPDATE to say what changes.
DELETE FROM
Remove rows. Deletes row data, usually with a WHERE filter for safety.

Changing Structure

The statements that define, alter, or remove the table itself.

CREATE TABLE
Make a new table. Define the table and its columns.
ALTER TABLE
Change a table. Add, drop, or rename parts of the schema.
ADD
Add a column. Commonly used with ALTER TABLE.
DROP
Remove something. Can remove a column or a whole table depending on context.
RENAME TO
Rename a table. Change the table name without rebuilding it.
DROP TABLE
Delete the whole table. Stronger than deleting rows because the structure goes too.
IF EXISTS / IF NOT EXISTS
Guard clauses. Avoid errors when dropping or creating tables conditionally.

Table Rules And Data Types

These are schema words rather than query words, but the later lessons introduce them.

INTEGER
Whole number type. For values like counts or ids.
VARCHAR(...)
Variable-length text. Text up to a chosen maximum size.
DATETIME
Date and time type. Stores time-based values.
PRIMARY KEY
Main unique identifier. The column that uniquely identifies each row.
AUTOINCREMENT
Auto-numbering id. Often used for a primary key that grows automatically.
NOT NULL
Value required. This column is not allowed to be missing.
DEFAULT
Fallback value. Used when no value is supplied.
CHECK (...)
Validation rule. Limits what values are allowed.
FOREIGN KEY
Link to another table. A column whose values should match keys in another table.

Punctuation And Small Symbols

These are tiny, but they cause a surprising amount of friction when you are new.

;
End of query. Treat it as the normal full stop for a SQL statement.
,
Separator. Splits columns, values, or expressions in a list.
( )
Group parts together. Used in functions, inserted values, and subqueries.
/ * + -
Math operators. Used in expressions when SQL is computing a value.

Difficulty Jumps

Text versus number filtering

This becomes difficult because the question feels the same, but SQL expects different syntax depending on the data type.

  • What goes wrong: you write text as if it were a number, or forget quotes.
  • Why it happens: both are filtered with WHERE, so they look deceptively similar.
  • How to solve it: numbers stay bare, text gets single quotes.
Look at this: WHERE year = 2004 versus WHERE club = 'Liverpool'.

`LIKE` with `%` and `_`

This is hard because you are no longer asking for an exact value. You are asking for a text pattern.

  • What goes wrong: learners try to use = with a wildcard, such as = 'WALL-%'.
  • Why it happens: the eye notices the text first, not the fact that the question has changed from exact match to pattern match.
  • How to solve it: use = for exact known text and LIKE only when the shape matters.
Look at this: exact title: WHERE title = 'WALL-E'. Pattern: WHERE title LIKE 'WALL-%'.

`INNER JOIN` versus outer joins

The syntax is not the real difficulty. The real difficulty is deciding what should happen to rows with no match.

  • What goes wrong: you use an inner join and accidentally lose rows you still cared about.
  • Why it happens: beginners often assume “combine tables” means “keep everything.”
  • How to solve it: ask one question first: should unmatched rows vanish or remain visible?
Look at this: INNER JOIN keeps matches only. LEFT JOIN keeps all rows from the left table and fills the missing side with NULL.

`NULL` logic

NULL looks like a blank value, but SQL treats it as missingness, not as ordinary text or number data.

  • What goes wrong: learners try = NULL or assume blank means zero or empty text.
  • Why it happens: other languages train you to compare values directly.
  • How to solve it: switch mental model and treat missingness as its own condition.
Look at this: not WHERE building_id = NULL, but WHERE building_id IS NULL.

Row thinking to group thinking

Aggregates feel strange because you stop asking about individual rows and start asking summary questions.

  • What goes wrong: you expect the query to show detail rows and a summary at the same time without grouping properly.
  • Why it happens: the learner is still thinking “show me records,” not “summarise categories.”
  • How to solve it: ask whether the output should be one row per record or one row per group.
Look at this: SELECT director, COUNT(*) FROM movies GROUP BY director; returns one row per director, not one row per movie.

`WHERE` versus `HAVING`

This is a classic confusion because both clauses feel like filters, but they act at different stages of the query.

  • What goes wrong: learners try to filter an aggregate with WHERE.
  • Why it happens: the grouped result does not exist yet when WHERE runs.
  • How to solve it: remember the split: rows first with WHERE, groups later with HAVING.
Look at this: WHERE year > 2000 filters rows. HAVING COUNT(*) > 2 filters grouped results.

Lessons 1 To 3 Sample

Lesson 1

What am I asking for?

Which parts of each row do I want back?

  • `SELECT` chooses properties.
  • `FROM` names the table.
  • The answer is a returned table.
SELECT title FROM movies; SELECT title, year FROM movies;
Lesson 2

Which rows stay?

Out of the whole table, which rows should remain in the result?

  • `WHERE` tests each row.
  • Matching rows stay.
  • Non-matching rows disappear.
WHERE year = 2003 WHERE year BETWEEN 2000 AND 2010
Lesson 3

How do I filter text?

What if I know the text exactly, only partly, or only by a pattern?

  • Text values need quotes.
  • `=` and `LIKE` are not the same question.
  • `%` means any run of characters.
WHERE director = 'John Lasseter' WHERE title LIKE 'Toy Story%'

Big Shifts Summary

Shift 01

From table to question

You stop treating SQL as "opening data" and start using it to ask a precise question.

Shift 02

From rows to relationships

You move from one table at a time to matching related records across tables.

Shift 03

From detail to summary

You stop only asking what happened row by row and start asking how much, how many, and by category.
